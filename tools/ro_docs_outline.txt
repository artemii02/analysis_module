P0131 [normal]: Серверная часть приложения функционирует на базе Java и Spring Boot, использует PostgreSQL для хранения данных, JWT для аутентификации, а также интеграцию с внешним AI-сервисом оценивания.

P0137 [Heading 1]: ВЫПОЛНЕНИЕ ПРОГРАММЫ
P0138 [Heading 2]: Подготовка к работе
P0139 [normal]: Перед началом эксплуатации оператор должен убедиться, что для выбранного контура заданы корректные значения доменных имен, секретов, параметров подключения к PostgreSQL, JWT-секрета, почтового аккаунта и параметров интеграции с внешним сервисом оценивания.
P0140 [normal]: В Kubernetes-контуре эти параметры задаются через values-файлы Helm chart и секреты, используемые chart'ами backend, frontend и database. Для внешней публикации сервисов должны быть настроены ingress-ресурсы и ClusterIssuer Let's Encrypt.
P0141 [Heading 2]: Локальный запуск
P0142 [normal]: Для локального запуска серверной части необходимо перейти в каталог backend-репозитория и выполнить команду docker compose up --build. В результате будут подняты контейнеры PostgreSQL и backend-приложения, публикуемого на порту 8080.
P0143 [normal]: Для локального запуска frontend-части необходимо установить зависимости, при необходимости сгенерировать клиент по OpenAPI-описанию и выполнить команду npm run dev. После этого интерфейс будет доступен по локальному адресу разработки, а backend API должен быть доступен отдельно.
P0144 [normal]: Если интеграция с внешним AI-сервисом недоступна, оператор может использовать локальный контур в режиме без генерации отчетов либо с тестовыми параметрами интеграции, заданными через переменные окружения.
P0145 [Heading 2]: Развертывание в Kubernetes-контуре
P0146 [normal]: Production-развертывание проекта организовано через Helmfile. Базовый сценарий развертывания заключается в переходе в каталог deploy-репозитория, заполнении values.yaml и выполнении команды helmfile sync.
P0147 [normal]: Общий файл оркестрации включает релизы cert-manager, ingress-nginx, postgres, backend, frontend, prometheus, loki, promtail, tempo, otel-collector и grafana. Порядок развертывания зависимостей задается через needs и обеспечивает корректный ввод приложений в эксплуатацию.
P0148 [normal]: После первичного развертывания cert-manager оператор должен применить манифест ClusterIssuer. Затем необходимо проверить выпуск TLS-сертификатов и доступность frontend и backend по HTTPS-адресам.
P0149 [normal]: Для выборочного обновления отдельных подсистем допускается выполнение команд helmfile -l app=backend sync, helmfile -l app=frontend sync, helmfile -l app=postgres sync и helmfile -l stack=monitoring sync.
P0150 [Heading 2]: Автоматизированное обновление через CI/CD
P0151 [normal]: В проекте реализован CI/CD на базе GitHub Actions. Отдельные release-workflow для backend и frontend собирают Docker-образы, публикуют их в Docker Hub и автоматически обновляют image tag в deploy-репозитории.
P0152 [normal]: После изменения deploy-репозитория workflow развертывания по push в ветку main определяет изменившиеся стеки и запускает выборочный helmfile sync на целевом сервере по SSH. Поэтому в штатном режиме оператор контролирует корректность прохождения pipeline и вмешивается вручную только при сбое автоматического сценария.
P0153 [normal]: Для ручного повторного запуска допускается использование workflow_dispatch в GitHub Actions либо прямой запуск команд helmfile на сервере сопровождения при наличии необходимых прав доступа.
P0154 [Heading 2]: Контроль работоспособности
P0155 [normal]: После запуска или обновления оператор должен проверить готовность pod'ов, сервисов и ingress-ресурсов командами kubectl get pods -A, kubectl get svc -A и kubectl get ingress -A. При необходимости следует использовать kubectl logs и kubectl rollout status для диагностики и контроля завершения развертывания.
P0156 [normal]: Серверная часть экспортирует health-пробы и метрики через actuator-endpoint'ы, а наблюдаемость реализована через Prometheus, Grafana, Loki, Tempo и OTel Collector. При возникновении ошибок оператор должен проверить dashboards, application logs и trace-данные.
P0157 [normal]: Особое внимание следует уделять доступности PostgreSQL, корректности CORS-настроек, состоянию TLS-сертификатов, а также доступности внешнего AI-сервиса, поскольку задержки или отказ интеграции напрямую влияют на генерацию итогового отчета.

P0166 [Heading 2]: Работа оператора в административном контуре

P0168 [normal]: Для ускоренного наполнения контента предусмотрен импорт JSON-файла через административный раздел. Перед публикацией нового профиля оператор должен убедиться, что профиль содержит описание, направление, уровень и требуемый набор вопросов в корректном порядке.
P0170 [Heading 1]: СООБЩЕНИЯ ОПЕРАТОРУ
P0171 [normal]: В процессе эксплуатации оператор может получать сообщения пользовательского интерфейса, API и инфраструктурного контура. Ниже приведены типовые сообщения и рекомендуемые действия.
P0172 [normal]: "Требуется регистрация" - пользователь пытается начать собеседование без авторизации. Действие оператора: предложить зарегистрироваться или выполнить вход.
P0173 [normal]: "Получаем первый вопрос..." - система инициировала старт сессии и ожидает ответ backend. Действие оператора: дождаться завершения запроса; при зависании проверить backend logs и доступность базы данных.
P0174 [normal]: "Генерируем отчёт..." или "Интервью завершено. Готовим отчёт..." - backend обрабатывает завершенную сессию и ожидает результат от AI-сервиса. Действие оператора: контролировать состояние PROCESSING, логи backend и доступность внешней интеграции.
P0175 [normal]: "Invalid email or password" - введены неверные учетные данные. Действие оператора: проверить правильность email и пароля, при необходимости использовать процедуру сброса пароля.
P0176 [normal]: "User with this email already exists" - попытка повторной регистрации на уже существующий адрес. Действие оператора: предложить вход в систему или восстановление доступа.
P0177 [normal]: "Invalid or expired refresh token" и "Invalid or expired password reset token" - истек срок действия токена. Действие оператора: инициировать повторную авторизацию или повторный запрос на сброс пароля.
P0178 [normal]: "Only in-progress sessions can be paused", "Only paused sessions can be resumed", "Cannot send message for the current session state" - выполнена операция, не соответствующая текущему состоянию сессии. Действие оператора: проверить текущее состояние и повторить допустимую команду.
P0179 [normal]: "External assessment service is unavailable. Please try again later." или сообщение о временной недоступности circuit breaker - внешний AI-сервис недоступен. Действие оператора: проверить сетевую связанность, параметры интеграции и повторить генерацию позже.
P0180 [normal]: "Validation failed" - запрос содержит неполные или некорректные данные. Действие оператора: исправить обязательные поля формы или импортируемого файла и повторить операцию.
P0181 [normal]: HTTP 5xx и сообщение "Unexpected server error" - внутренняя ошибка backend. Действие оператора: изучить application logs, трассировки и состояние зависимых сервисов, после чего выполнить повторный запрос или перезапуск deployment.

P0090 [Heading 2]: 1.2 Краткая характеристика области применения программы
P0091 [Normal]: Программа является автономным серверным модулем в составе веб-приложения для имитации технических собеседований. Модуль принимает от основного backend-приложения данные завершенной сессии: параметры сценария, тексты вопросов, ответы пользователя и служебные метаданные. На основании этих данных модуль формирует структурированный отчет с оценками, си
P0092 [Normal]: Модуль предназначен для образовательного и карьерного применения: подготовки кандидатов к собеседованиям, проведения учебных тренировок, анализа качества ответов по направлениям Backend, Frontend и DevOps для уровней Junior и Middle. Пользовательский интерфейс и выбор вопросов находятся в зоне ответственности основного приложения; данный модуль отв
P0093 [Normal]: Основной рабочий режим использует локальную большую языковую модель Qwen/Qwen2.5-3B-Instruct и LoRA/QLoRA-адаптер, обученный на русскоязычном датасете интервью. Дополнительно предусмотрены режимы mock для быстрой проверки инфраструктуры и ollama для альтернативного локального запуска.

P0182 [Heading 1]: 4 СООБЩЕНИЯ ОПЕРАТОРУ
P0183 [Heading 2]: 4.1 Общий формат ответа об ошибке
P0184 [Normal]: При ошибке модуль возвращает структурированный JSON-ответ. Формат ответа:
P0185 [Normal]: { "status": "error", "errorCode": "INVALID_INPUT", "message": "Описание ошибки", "details": {} }
P0186 [Normal]: Код ошибки используется backend-приложением для выбора сценария обработки, а поле details применяется для диагностики.
P0187 [Heading 2]: 4.2 OK и READY
P0188 [Normal]: Статусы ok и ready означают, что операция выполнена успешно. Для health-запроса status=ok означает доступность сервиса. Для отчета status=ready означает, что отчет сформирован и доступен в поле report.
P0189 [Heading 2]: 4.3 CREATED и PROCESSING
P0190 [Normal]: Статусы created и processing применяются к асинхронным задачам. created означает, что задача зарегистрирована. processing означает, что анализ выполняется. Оператор должен дождаться перехода задачи в ready или error.
P0191 [Heading 2]: 4.4 AUTHENTICATION_ERROR
P0192 [Normal]: Ошибка возникает при отсутствии заголовка X-API-Key или при передаче неверного API-ключа. Оператор должен проверить значение ANALYSIS_API_KEY и настройки вызывающего backend-приложения.
P0193 [Heading 2]: 4.5 INVALID_INPUT
P0194 [Normal]: Ошибка означает, что запрос не прошел валидацию. Типичные причины: отсутствуют requestId, sessionId, clientId; отсутствует questionText или answerText; превышено ограничение 20 вопросов в сессии; превышена длина ответа 4000 символов.
P0195 [Heading 2]: 4.6 REQUEST_CONFLICT
P0196 [Normal]: Ошибка возникает, если один и тот же requestId повторно используется с другим телом запроса. Оператор должен передавать новый requestId для новой сессии или повторять исходный запрос без изменений.
P0197 [Heading 2]: 4.7 JOB_NOT_FOUND и REPORT_NOT_READY
P0198 [Normal]: JOB_NOT_FOUND означает, что задача с указанным jobId отсутствует в хранилище. REPORT_NOT_READY означает, что отчет еще формируется. Оператор должен проверить корректность jobId и состояние хранилища задач.
P0199 [Heading 2]: 4.8 INVALID_MODEL_OUTPUT, MODEL_TIMEOUT и INTEGRATION_ERROR
P0200 [Normal]: Эти ошибки относятся к LLM-компоненту. INVALID_MODEL_OUTPUT означает, что модель вернула невалидный JSON. MODEL_TIMEOUT означает превышение допустимого времени генерации. INTEGRATION_ERROR означает сбой взаимодействия с внешним или локальным LLM-provider. Оператор должен проверить журналы analysis-module, параметры токенов генерации, режим LLM и до
P0201 [Heading 2]: 4.9 DATABASE_UNAVAILABLE и DATABASE_WRITE_ERROR
P0202 [Normal]: Ошибки базы данных возникают при недоступности PostgreSQL, неверной строке подключения или невозможности записать статус задачи. Оператор должен проверить контейнер postgres, переменную ANALYSIS_DATABASE_URL и журналы базы данных.
P0203 [Heading 2]: 4.10 INTERNAL
P0204 [Normal]: INTERNAL означает непредвиденную внутреннюю ошибку. Оператор должен сохранить requestId, jobId, время ошибки, фрагмент журнала и передать данные разработчику для анализа.

P0228 [Normal]: Backend - серверная часть основного приложения, которая проводит интервью, хранит вопросы и передает завершенную сессию в модуль анализа.

P0232 [Normal]: PostgreSQL - система управления базами данных, используемая для хранения статусов задач и сформированных отчетов.
P0233 [Normal]: Сессия - набор вопросов и ответов пользователя в рамках одного интервью. В продуктовой версии сессия содержит 20 вопросов.
P0234 [Normal]: Рубрика - набор критериев и весов оценки ответа: корректность, полнота, ясность, практичность и терминология.
